# HFT Order Book Matching Engine + Kalman Filter Latent Price Recovery

A C++ limit order book matching engine paired with a 1D Kalman filter that recovers the latent "true" price from noisy mid-quote observations contaminated by bid-ask bounce. Validated against synthetic ground truth.

## Headline numbers

**C++ Matching Engine**
- 3.46M events/sec book-only throughput
- 1.82M events/sec wall throughput (end-to-end with generator)
- p50 = 200ns, p99 = 1µs, p99.9 = 2.2µs per-event latency
- 15/15 unit + matching tests passing

**Kalman Filter**
- 16.7% MSE reduction vs raw mid-price baseline
- Steady-state Kalman gain K = 0.14
- Matches best-tuned EMA (α = 0.15) to within 0.13% MSE (confirms theoretical equivalence at steady state under stationary noise)
- Validated directly against synthetic ground truth — only possible because data is generated by our own simulator

## What this project demonstrates

Three skills in one cohesive piece:

1. **Systems engineering** — low-latency C++ matching engine with explicit data structure justifications (std::list for stable iterators enabling O(1) cancel-by-ID, hash-indexed locators, percentile latency measurement).
2. **Statistical modeling** — 1D Kalman filter derived from Bayes' rule and Gaussian conjugacy on paper, implemented from scratch in NumPy.
3. **Experimental rigor** — synthetic data with known ground truth enables direct quantitative validation, impossible with real exchange data where the latent price is unobservable.

## How to build and run

### C++ engine

Requires CMake 3.16+ and a C++17 compiler (GCC 16.1 / Clang 14 / MSVC 2019+).

```bash
cd cpp/build
cmake -G "MinGW Makefiles" -DCMAKE_BUILD_TYPE=Release ..   # or your generator
cmake --build .

# Run all 15 tests
./test_orderbook

# Run a simulation (writes data/midprices.csv, data/trades.csv)
./orderbook --events 100000 --output ../../data/

# Benchmark throughput and latency (writes benchmark_latencies.csv)
./benchmark --events 1000000 --warmup 100000
```

### Python analysis

Requires Python 3.10+.

```bash
cd python
pip install -r requirements.txt

# Run filter tests
python test_kalman.py

# Open the notebook
jupyter lab analysis.ipynb
```

Run all cells. The notebook produces overlay plots, the variance reduction comparison, and the latency histogram.

## Design decisions

### C++ engine

- **`std::list` for FIFO at each price level** — chosen over `std::deque` so the order-lookup table can store stable list iterators. This enables O(1) cancel-by-ID anywhere in the queue.
- **`std::map<int64_t, PriceLevel>` per side** — bids use `std::greater`, asks use default `std::less`, so `begin()` always returns the best price on either side. O(1) best-price access, O(log L) inserts where L is the active price-level count.
- **`std::unordered_map<id, OrderLocator>`** — hash-indexed locator stores `{side, price, list iterator}` so cancels jump directly to the relevant list node without searching.
- **Integer ticks (`int64_t`)** — prices stored as cents, eliminates float rounding errors at the matching layer.
- **Two-method submit API** — `submitLimitOrder` and `submitMarketOrder` share a private `matchOrder(stop_on_price: bool)` helper. Avoids an `OrderType` field on the Order struct.
- **`isResting()` public accessor** — added specifically to fix a partial-fill bug in the resting-id tracker: without this check, partially-consumed passive orders would be incorrectly unregistered.
- **Single-threaded by design** — deterministic execution order, no lock overhead. Production matching engines are single-threaded for the same reason; scale comes from one-thread-per-symbol.

### Synthetic event generator

- **52% LIMIT / 40% CANCEL / 8% MARKET** — calibrated to real exchange flow ratios.
- **Brownian latent price** — true price drifts via Gaussian steps σ=1 per event. Bid-ask bounce is *emergent* from market orders interacting with the resting book, not an explicitly added noise term.
- **Limit prices biased to passive side** — Gaussian offset around true price, signed by side, so most limits rest and the book builds realistic depth.
- **CANCEL → LIMIT fallback** — when the resting-id tracker is empty (early simulation), cancels fall back to limits. Self-stabilizing bootstrap.
- **Resting-id tracking with O(1) swap-and-pop** — generator maintains a vector of resting orders synced to book state, enabling uniform-random cancel selection.

### Benchmark methodology

- **Live event generation in the timed loop** — selective bracketing of only the book operation so `gen.next()` cost is not attributed to book latency. An earlier pre-generation design broke the cancel-tracker (book bloated to 33K orders, throughput crashed to 0.55M evt/s) — fixed by switching to live generation.
- **Warmup phase (100K events)** — ensures the book is in steady state when timing starts.
- **Percentile-based reporting** — p50, p90, p99, p99.9, p99.99 latency. Tail latency matters more than mean in HFT; mean hides scheduling outliers.
- **Two throughput numbers** — book-only (sum of measured per-event latencies, engine's intrinsic speed) and wall (full pipeline with generator overhead).

### Kalman filter

- **1D state, 1D observation** — multivariate state would extend to (price, velocity, order imbalance) but is out of scope here.
- **Constant Q and R** — assumes stationary noise. Real markets have regime-changing noise; the natural extension is the EM algorithm to learn (Q, R) jointly with state estimates.
- **Predict and update implemented by hand** — math derived from Bayes' rule + Gaussian conjugacy on paper before coding. The closed-form solution makes the filter directly verifiable.
- **Validated by tests** — six unit tests covering predict mechanics, update mechanics, Kalman gain limits (K=0 and K=1), and end-to-end filter behavior on a known random walk.

## Methodological notes

### Why synthetic data, not real exchange feeds

The standard answer is "we want ground truth." With real LOBSTER or ITCH data, the latent fair value of the asset is unobservable — you can compute filtered estimates but can't directly measure how well they recover the truth. By generating data with a known latent price, we can quantify the filter's performance with MSE-vs-truth and other direct metrics.

The trade-off: our synthetic data has stationary noise, while real markets have regime changes, volatility clustering, and other non-stationary features. The natural workflow is to validate on synthetic data (as we do here) and apply to real data with regime-adaptive extensions for the real-world case.

### R-estimation challenges (a real finding, not a bug)

We tried three estimators for the observation noise variance R:
1. **Naive variance of consecutive differences** (½ Var(Δy)) — gave R̂ ≈ 7
2. **Residual from rolling local mean** — gave R̂ from 12 to 55 depending on window size, with no plateau
3. **Roll's (1984) microstructure estimator** (negative autocovariance of Δy) — gave R̂ ≈ 4

The ground truth R is 42.5. All three estimators were significantly biased.

The cause: **our observation schedule writes one mid-price per event, but the mid only changes on ~17% of events** (limit/cancel/market events affecting the touch). The 83% of observations where mid is unchanged violate the assumption of independent bounce information that all three estimators rely on.

The principled fix is the EM algorithm, which jointly learns (Q, R) from data without assuming any particular observation schedule. This is the main stretch goal for the next iteration. For the analysis here we use R = 42 (the ground-truth value) to demonstrate the filter's performance with correctly-specified parameters, documenting the parameter-estimation challenge as a finding rather than glossing over it.

### Why the Kalman filter matches EMA performance here

At steady state with constant Q and R, the Kalman filter converges to a fixed gain K* = P* / (P* + R) and is mathematically equivalent to an EMA with α = K*. We confirmed this: the best EMA (α=0.15) matches the Kalman filter (steady-state K=0.14) to within 0.13% MSE.

The Kalman framework's advantages over EMA emerge in three settings we don't exhibit:
1. **Time-varying Q or R** — Kalman adapts gain online; EMA's α is fixed.
2. **Explicit uncertainty tracking** — Kalman provides Pₜ⁺ at every step.
3. **Multivariate state** — Kalman generalizes to vector state-space models.

Implementing Kalman from first principles establishes the foundation for those extensions.

### Forecasting limitations

The filter is a smoother/state estimator — best guess of the *current* state, not a forecaster. Under a random-walk model, the optimal k-step forecast equals the current estimate, with uncertainty growing as √(k·Q). The filter has no predictive power beyond the current value, which is the correct behavior under EMH-like assumptions.

Genuine predictive capability would require richer state-space models: 2D state with velocity/drift, microstructure features (order imbalance, trade-direction history), or regime-switching dynamics. These are natural extensions and are listed under future work.

### Directional accuracy as a separate metric

Beyond MSE, we measured the fraction of steps where each estimator correctly predicted the *sign* of the true price change. On steps where both estimator and truth move:
- Raw mid: 55.7% directional accuracy (slightly better than random)
- Kalman: 51.0% directional accuracy (essentially random)

This is not a contradiction with the MSE result. The Kalman filter optimizes for squared-error minimization, not next-tick direction prediction. The raw mid only updates when a real market event reshapes the touch (~17% of observations), so when it does move, the move correlates with a real underlying change. The filter trades directional sharpness for continuous smoothness — a meaningful design choice for downstream applications like VWAP/TWAP execution that benefit from continuous fair-value estimates.

## Future work

In rough order of impact:

1. **EM algorithm** for joint Q, R learning from data — addresses the parameter-estimation difficulties documented above.
2. **Regime-switching Kalman / Hamilton filter** — handles non-stationary noise (intraday volatility regimes, event-driven moves).
3. **Multivariate state** — extend to (price, velocity, order imbalance) for predictive (not just smoothing) capabilities.
4. **Real exchange data** — parse LOBSTER or ITCH and validate the filter's behavior on real microstructure (without ground truth, using indirect diagnostics like residual whiteness and predictive MSE).
5. **Custom price-ladder array** instead of `std::map` — eliminates the O(log L) factor in matching for further latency improvement.
6. **Execution algorithm overlays** — VWAP, TWAP, implementation shortfall — built on the matching engine as separate strategy modules.
